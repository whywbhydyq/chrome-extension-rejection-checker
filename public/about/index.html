About this local Chrome extension checker

Last updated: June 6, 2026

Chrome Extension Rejection Checker is a free browser-based preflight tool for developers preparing Manifest V3 extension ZIP files for Chrome Web Store review. It is designed for one specific workflow: before uploading a ZIP to the Chrome Web Store Developer Dashboard, run a local package scan and identify common technical risks that can cause rejection, reviewer questions, or avoidable resubmission cycles.

The tool is part of the YmirTool collection and is maintained as an independent developer utility. It is not affiliated with Google, Chrome, Chromium, or the Chrome Web Store review team. Its purpose is to make public Chrome extension guidance easier to apply to a concrete package, not to replace official documentation or policy review.

What the scanner does locally

Package structure

It checks whether the ZIP contains a root-level manifest.json, whether referenced files exist, and whether the package looks nested in a way that can break upload expectations.

Manifest V3 risks

It reads Manifest V3 fields such as permissions, host permissions, icons, background service worker references, content scripts, web accessible resources, sandbox pages, and content security policy.

Code pattern signals

It scans submitted text files for remote hosted code patterns, dynamic script loading, eval, new Function, string timers, WebAssembly loaders, worker URLs, and other review-sensitive patterns.

Privacy review prompts

It highlights broad host permissions, sensitive Chrome APIs, externally connected URLs, and disclosure reminders that developers should compare with their Developer Dashboard privacy fields.

What makes the guidance trustworthy

The guidance is grounded in public Chrome extension and Chrome Web Store documentation, then mapped to static package checks that can be performed in a browser without uploading the extension. Each core guide links back to source documentation so developers can verify the reasoning behind a warning. The scanner intentionally separates hard package problems from softer review reminders because Chrome Web Store decisions can depend on runtime behavior, listing metadata, reviewer judgment, account history, and policy context outside the ZIP.

What the checker cannot guarantee

  • It cannot guarantee Chrome Web Store approval or predict manual reviewer judgment.
  • It cannot read Developer Dashboard fields, privacy practice declarations, support URL settings, promotional assets, or reviewer notes you have not provided.
  • It cannot execute the extension, inspect live network behavior, or validate server-side data handling.
  • It cannot know whether a requested permission is truly necessary for your product purpose.

How to use the report

Treat findings as a prioritized pre-submission checklist. Fix critical package and Manifest V3 issues first, compare policy reminders with official Chrome documentation, update your extension version, rebuild your production ZIP, and then review Developer Dashboard disclosures before resubmission. A clean local report is useful evidence that obvious package-level risks were checked, but it is still only a preflight result.
